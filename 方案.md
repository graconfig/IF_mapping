## 1. 目标

把 `历史设计书/` 下 44 份 SAP 接口设计书 (インタフェース項目定義書) 沉淀为结构化知识库，用于：

1. **新字段匹配**：当新接口有一个外部字段（日文项目名 / 技术名 / 类型 / 长度）时，快速查出在历史设计书中已经映射到的 SAP 字段（表 + 字段）。
2. **增量融入**：新来的参考文档能按同一流程入库，不破坏已有数据。
3. **会话延续**：`CLAUDE.md` 与 `knowledge/README.md` 让未来的对话能直接上手使用，而非每次重建上下文。

### 映射表的列布局（23 列，固定）

```
左块(連携元)  中间(変換)            右块(連携先)         右尾(补充)
0   №         6   №                 10  №                17  桁数
1   項目名称   7   変換仕様           11  項目名称          18  コード体系
2   構造       8   I/O               12  構造              19  補足・その他
3   技術名称   9   現行編集仕様       13  技術名称          20-22 調整メモ
4   文字数                            14  文字数
5   属性                              15  属性
```

### 关键发现

- **受信表**：数据流 外部 → SAP，SAP 在右侧
- **返信表**：数据流 SAP → 外部，**SAP 在左侧**（R4 表头的 `K-Warranty`/`部品SAP` 是模板残留，不可信）
- **SAP 侧识别**：靠"構造"列是否像 SAP 表名（大写+数字+下划线，如 `VBAK` / `ADRC` / `PRCD_ELEMENTS`），而不是靠表头文字

## 3. 架构

```
Excel 设计书 ──(extract_mapping.py)──▶  field_mappings.jsonl  (人可读, git 友好)
                                                 │
                                                 ▼
                                      (build_index.py)
                                                 │
                                                 ▼
                                          ifs.db  (SQLite, 多维索引)
                                                 │
                                                 ▼
                         新字段 ──(match_field.py)──▶  Top-N SAP 候选
```

**分工原则**：JSONL = 真源 (source of truth)，SQLite = 可重建的查询加速层。

---

## 4. 数据模型

每条字段映射记录（JSONL 一行）：

| 字段 | 说明 |
|---|---|
| `ifid` / `if_name` | IF 编号 / 名称 |
| `direction` | `inbound`（外部→SAP）/ `outbound`（SAP→外部） |
| `row_no` | 在原 Excel 中的行号（便于回溯） |
| `external_no` / `external_name` / `external_tech_name` / `external_length` / `external_attr` | 外部系统侧字段属性 |
| `sap_struct` / `sap_tech_name` | **SAP 表 + 字段**（如 `VBAK.KUNNR`） |
| `sap_name` / `sap_length` / `sap_attr` | SAP 字段日文名 / 长度 / 类型 (CHAR/NUMC/QUAN/...) |
| `sap_side_pos` | `left` / `right`，记录 SAP 在原表中位于哪一侧 |
| `conversion_spec` | 変換仕様 |
| `io` | I / O / I/O |
| `current_spec` | 現行編集仕様（参考） |
| `code_system` / `digits` / `note` / `adj_remark` | 补充信息 |
| `has_sap_mapping` | 是否有明确 SAP 字段（便于过滤） |
| `source_file` | 来源 Excel 文件名 |

## 7. 标准使用流程

### A. 查询"某外部字段 → SAP 字段"

```bash
python3 scripts/match_field.py --name 得意先コード
python3 scripts/match_field.py --tech KUNNR
python3 scripts/match_field.py --name 品番 --length 10 --ifid IFZ9000003
python3 scripts/match_field.py --name 納所 --direction inbound
```

输出：Top-N 候选，含评分、命中原因、SAP 表.字段、変換仕様、同字段出现过的其他 IFID。

### B. 增量新增一份设计书

```bash
# 1. 放到 历史设计书/ 或 参考资料/ 下
# 2. 追加抽取
python3 scripts/extract_mapping.py 历史设计书/新文件.xlsx \
    -o knowledge/field_mappings.jsonl --append
# 3. 重建索引
python3 scripts/build_index.py
```

### C. 从零重建

```bash
python3 scripts/extract_mapping.py 历史设计书/*.xlsx -o knowledge/field_mappings.jsonl
python3 scripts/build_index.py
```

## 8. 已确认的设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 抽取范围 | 只抽 `項目マッピング(受信/返信)`，暂不抽 `IFレイアウト`/协议表 | 字段匹配的最小必要集，后续可加 |
| 存储形式 | JSONL 作真源 + SQLite 作索引 | JSONL 可 diff / 复核；SQLite 可丢了重建 |
| SAP 侧识别 | 看"構造"列是否像 SAP 表名（正则） | 比信任表头文字可靠（返信表的模板有残留错误） |
| 执行节奏 | 先跑 1 份 pilot 验收，再批量 | 在全量前验证抽取与识别逻辑的正确性 |

## 9. 已知风险 / 待解

1. **模板分化**：44 份设计书若列布局不完全一致（比如某些老版本列偏移），当前 `extract_mapping.py` 里硬编码的列索引 (0..22) 会误取。**缓解**：批量跑完后检查 `[error]` 日志与 `has_sap_mapping=1` 计数，异常少的文件人工点开看。
2. **多字段合并行**：已观察到类似 `ADRC\nADRC.NAME1\nNAME2`（一行映射到两个 SAP 字段）。当前按原样保留字符串。**缓解**：全量后统计出现频次，决定是否拆成两条记录。
3. **同一外部字段在不同 IF 有不同 SAP 映射**：是正常业务现象（不同场景本就映射到不同字段），当前设计保留所有变体，查询时用 `_also_seen_in` 列出来源。
4. **コード体系/桁数列大量为空**：只有少数字段填了，目前作为可选字段保留，不影响主流程。
